# AI 활용 기록

목적: 프로젝트를 만들면서 AI를 어떤 방식으로 활용했는지 기록한다. 단순히 AI를 사용했다는 사실보다, 어떤 문제를 해결하기 위해 어떤 방식으로 활용했고 어떤 산출물이 생겼는지를 남긴다.

---

## 1. 프로젝트 방향 정리

문제:

- 기술블로그 프로젝트의 목표, MVP 범위, 확장 방향이 한 문서 안에서 명확히 정리되어 있지 않았다.
- 블로그, 이력서, 관리자, 모노레포, 모바일 앱 같은 아이디어가 섞여 있어 어떤 기능을 먼저 만들어야 하는지 판단하기 어려웠다.

AI 활용 방식:

- 대화형 AI를 사용해 프로젝트 목표와 MVP 단계를 나눴다.
- 1차, 2차, 3차 MVP와 후순위 확장을 구분해 문서 구조를 정리했다.

산출물:

- `README.md`
- `docs/01-blog-project-roadmap.md`
- `docs/02-study-blogging-roadmap.md`
- `docs/03-supabase-erd-design.md`

검증 / 수정:

- 1차 MVP에 필요한 페이지를 `/`, `/resume`, `/blog`, `/blog/[slug]`로 다시 좁혔다.
- 관리자, Supabase, 모노레포, 모바일 앱은 후순위로 이동해 초기 범위가 커지지 않도록 수정했다.
- 문서에서 오래된 라우트, 페이지 이름, 프로젝트 목적 표현이 남아 있는지 검색해 정리했다.

선택 이유:

- MVP 범위를 먼저 고정하면 구현 속도가 빨라진다.
- 문서 간 용어를 통일하면 이후 유지보수가 쉬워진다.
- 기능을 단계별로 나누면 지금 해야 할 작업과 나중에 할 작업을 구분하기 쉽다.

배운 점:

- 프로젝트 초반에는 기능을 많이 적는 것보다 하지 않을 일을 정하는 것이 더 중요하다.
- README, 개발 로드맵, 학습 로드맵이 같은 방향을 말해야 실제 개발 우선순위가 흔들리지 않는다.

## 2. Plan 모드를 활용한 로드맵 정리

문제:

- 초기 로드맵에는 필요한 작업과 불필요한 설명이 섞여 있어 바로 개발 체크리스트로 쓰기 어려웠다.
- 디자인, 콘텐츠 모델, 라우팅, 배포, 테스트 같은 작업 순서가 분명하지 않았다.

AI 활용 방식:

- Plan 모드를 사용해 작업을 기능 단위로 나눴다.
- 페이지별 구현 항목과 완료 기준을 함께 정리했다.

산출물:

- `docs/01-blog-project-roadmap.md`

검증 / 수정:

- 와이어프레임과 컴포넌트 목록처럼 당장 구현에 덜 필요한 항목은 줄였다.
- 프로젝트 생성, 공통 레이아웃, 홈, 이력서, 블로그 목록, 블로그 상세, 콘텐츠 데이터 구조 순서로 재정렬했다.
- Git, `.gitignore`, Tailwind, ESLint, Prettier, shadcn/ui, npm scripts 같은 초기 세팅 항목을 추가했다.

선택 이유:

- 기능 단위 체크리스트는 실제 개발 속도를 높인다.
- 완료 기준을 같이 두면 작업이 끝났는지 판단하기 쉽다.
- 로드맵을 구현 순서 중심으로 바꾸면 유지보수와 진행 상태 추적이 쉬워진다.

배운 점:

- 로드맵은 긴 설명보다 체크 가능한 작업 목록이어야 한다.
- 설계 문서는 개발을 미루기 위한 문서가 아니라 다음 행동을 명확히 하기 위한 문서여야 한다.

## 3. AI 퍼블리싱 스킬 제작

문제:

- 디자인을 Figma로 오래 설계하면 1차 MVP 개발이 늦어질 수 있었다.
- AI에게 퍼블리싱을 맡기더라도 매번 접근성, 모바일, semantic HTML 기준을 다시 설명해야 했다.

AI 활용 방식:

- 페이지 단위 퍼블리싱 프롬프트를 만들 수 있는 스킬을 제작했다.
- AI가 만든 UI 결과를 구조, 접근성, 모바일 기준으로 검증하도록 스킬 규칙을 보강했다.

산출물:

- `.codex/skills/blog-ui-publisher/SKILL.md`

검증 / 수정:

- AI가 생성한 퍼블리싱 구조에서 semantic tag 누락 가능성을 발견해 `header`, `nav`, `main`, `section`, `article`, `footer` 기준을 추가했다.
- 모바일 버튼 클릭 영역이 부족해질 수 있어 주요 액션은 최소 44px 높이를 우선하도록 기준을 넣었다.
- AI 결과를 바로 완료 처리하지 않고 `Validation Findings`, `Repair Notes`, `Accessibility Structure`를 출력하도록 수정했다.

선택 이유:

- Figma 설계보다 빠르게 MVP 화면 구현에 들어갈 수 있다.
- 반복되는 퍼블리싱 기준을 스킬로 고정하면 결과물의 일관성이 좋아진다.
- 접근성과 모바일 기준을 스킬에 넣어두면 검수 품질을 유지하기 쉽다.

배운 점:

- AI 퍼블리싱은 디자인을 생략하는 방식이 아니라, 구현에 필요한 기준을 작게 고정하고 빠르게 검증하는 방식이다.
- AI 결과를 그대로 쓰지 않고 구조와 접근성 기준으로 검증해야 실제 서비스 품질이 올라간다.

## 4. 커밋 메시지 스킬 제작

문제:

- 커밋 메시지 규칙은 README에 있었지만, 실제 변경 내용을 보고 일관된 메시지를 만들기 위한 절차는 없었다.
- AI가 만든 작업과 사용자가 직접 만든 작업을 커밋에서 구분하는 기준도 필요했다.

AI 활용 방식:

- README의 Conventional Commits 규칙을 바탕으로 커밋 메시지 생성 스킬을 만들었다.
- `git status`, staged diff, unstaged diff를 보고 변경 목적을 판단하도록 절차화했다.
- Codex가 직접 산출물을 작성하거나 수정한 경우에만 `Co-authored-by`를 제안하도록 기준을 추가했다.

산출물:

- `.codex/skills/commit-message-generator/SKILL.md`
- `README.md`
- `AGENTS.md`

검증 / 수정:

- 사용자가 직접 작업한 경우까지 `Co-authored-by`가 붙지 않도록 조건을 강화했다.
- 단순 질문, 가벼운 검토, 표현 확인, 낮은 가치의 보일러플레이트에는 공동 작성자 footer를 붙이지 않도록 수정했다.
- README, AGENTS, 스킬의 기준이 서로 다르지 않도록 문구를 맞췄다.

선택 이유:

- 커밋 메시지를 자동화하면 작성 속도가 빨라진다.
- 같은 기준으로 메시지를 만들면 Git 히스토리의 일관성이 좋아진다.
- AI가 만든 산출물과 사용자가 만든 산출물을 구분하면 기록의 신뢰도가 높아진다.

배운 점:

- 커밋 메시지 자동화는 단순히 문장을 만드는 것이 아니라 변경 목적을 분류하는 작업이다.
- AI 공동 작성 표기는 자동으로 붙이면 안 되고, 실제 산출물 작성 여부를 기준으로 판단해야 한다.

## 5. AI 활용 기록 작성 스킬 제작

문제:

- AI를 활용한 모든 작업을 기록하면 문서가 길어지고 핵심 경험을 찾기 어려워질 수 있었다.
- 어떤 작업을 기록해야 하는지 판단하는 기준이 필요했다.

AI 활용 방식:

- 기록할 가치가 있는 AI 활용 사례를 선별하는 스킬을 만들었다.
- 가치 있는 경우에만 문제, AI 활용 방식, 산출물, 검증/수정, 선택 이유, 배운 점으로 정리하도록 했다.

산출물:

- `.codex/skills/ai-usage-recorder/SKILL.md`
- `docs/04-ai-usage-log.md`

검증 / 수정:

- 단순 질문, 보일러플레이트 생성, 단순 번역, 포맷 변경은 기록하지 않도록 제외 기준을 추가했다.
- `검증 / 수정` 항목을 필수로 두어 AI 결과를 그대로 기록하지 않게 했다.
- 선택 이유에는 속도, 일관성, 유지보수, 구조 관점 중 하나 이상이 들어가도록 수정했다.

선택 이유:

- 기록할 가치가 있는 사례만 남기면 문서 유지보수가 쉬워진다.
- 같은 형식으로 기록하면 AI 활용 경험을 나중에 비교하기 쉽다.
- 문제 해결 중심으로 기록하면 포트폴리오 설명에 활용하기 좋다.

배운 점:

- AI 활용 기록에도 작성 기준과 제외 기준이 필요하다.
- 기록은 작업량을 보여주기보다 어떤 문제를 어떻게 해결했는지 보여줘야 한다.

## 6. GitHub 초기 연동

문제:

- GitHub quick setup 명령을 그대로 실행하면 기존 README에 불필요한 제목이 추가될 수 있었다.
- `main`, `dev`, 원격 저장소, `.gitignore` 기준을 초기 상태에서 정리해야 했다.

AI 활용 방식:

- AI에게 Git 초기화, 원격 저장소 연결, 초기 커밋, `main`/`dev` 브랜치 생성을 맡겼다.
- 기존 README를 유지하고 필요한 Git 명령만 골라 실행했다.

산출물:

- GitHub repository: `https://github.com/pigpgw/tech_blog.git`
- `main` 브랜치
- `dev` 브랜치
- `.gitignore`
- `AGENTS.md`

검증 / 수정:

- `git status`, `git branch -vv`, `git remote -v`, `git ls-remote`로 로컬/원격 상태를 확인했다.
- `.codex/skills`를 프로젝트에서 공유하기 위해 `.codex/`를 `.gitignore`에서 제거했다.
- 민감한 개인 설정이나 토큰은 `.codex`가 아니라 별도 로컬 환경에 두는 기준을 문서에 남겼다.

선택 이유:

- Git 초기 연결을 AI에게 맡기면 명령 순서 실수를 줄일 수 있다.
- 브랜치와 원격 상태를 바로 확인하면 이후 협업/배포 흐름을 유지하기 쉽다.
- `.codex/skills`를 버전 관리하면 프로젝트의 AI 작업 절차를 재사용할 수 있다.

배운 점:

- GitHub에서 제공하는 quick setup 명령도 현재 파일 상태에 맞게 골라 실행해야 한다.
- 공유할 AI 스킬과 민감한 로컬 설정은 명확히 구분해야 한다.
- AI가 직접 만든 산출물과 사용자가 직접 만든 산출물을 구분하기 위해 공동 작성자 기준이 필요하다.
