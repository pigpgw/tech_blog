## 이벤트가 너무 자주 발생하는 문제

검색 입력, 스크롤, 창 크기 변경 이벤트는 사용자가 한 번 조작해도 짧은 시간에 여러 번 발생한다. 이 이벤트를 모두 처리하면 불필요한 계산이 늘고 렌더링도 자주 일어난다. 특히 검색 API 호출처럼 비용이 있는 작업은 이벤트 수만큼 요청이 쌓일 수 있다.

## 디바운스

디바운스는 마지막 이벤트 이후 일정 시간이 지났을 때 한 번만 실행하는 방식이다. 사용자가 입력을 멈춘 뒤 검색하거나, 자동 저장을 지연시키거나, 필터 조건 변경이 끝났을 때 목록을 갱신하는 상황에 잘 맞는다.

## 쓰로틀

쓰로틀은 일정 시간 동안 최대 한 번만 실행한다. 스크롤 위치 표시, 드래그 좌표 계산, 무한 스크롤 감지처럼 연속적인 반응은 필요하지만 모든 이벤트를 처리할 필요는 없는 경우에 적합하다.

## 선택 기준

입력이 끝난 뒤 한 번만 처리하면 된다면 디바운스를 선택한다. 움직이는 동안 주기적으로 상태를 반영해야 한다면 쓰로틀을 선택한다. 두 방식 모두 이벤트를 막는 기술이 아니라 처리 밀도를 조절하는 전략이다.
